# RAG 知识库问答设计

## 目标

在当前 Streamlit 聊天 Demo 中增加一套可实际使用的本地知识库问答能力，支持用户上传 `Markdown / TXT / PDF` 文件，为当前会话绑定知识文件，在提问时先检索相关片段，再把检索结果和引用来源一并注入现有聊天链路。

本阶段采用“双模式”检索方案：

- 默认模式：本地向量化，可开箱即用，不依赖额外外部接口。
- 增强模式：如果用户额外配置了 Embedding 接口，则优先使用远端 Embedding 提升召回质量。

## 范围

本阶段包含：

- 上传 `Markdown / TXT / PDF` 文件
- 解析文本并切分为可检索片段
- 为片段生成向量并建立本地索引
- 把知识文件与当前会话关联
- 提问时执行检索并拼接引用来源
- 回答区展示引用来源
- 检索不到结果时自动退回普通聊天

本阶段不包含：

- 多用户权限隔离
- 云对象存储
- `pgvector` 或外部向量数据库
- 复杂 rerank、混合检索、多路召回
- 对 PDF 复杂表格、图片、版面结构的高保真恢复
- Agent 化工作流编排

## 设计原则

1. 先做“可用且可维护”的最小完整链路，不提前引入独立后端服务。
2. 尽量贴合当前项目的模块化风格，把 RAG 逻辑放入独立服务层。
3. 本地模式必须在不额外配置 Embedding 服务的情况下正常工作。
4. 远端 Embedding 只是增强路径，配置缺失、测试失败或运行失败时不能阻断本地模式。
5. 文件、索引和会话关联都要有明确生命周期，避免“文件删了但索引残留”的脏状态。

## 方案对比

### 方案一：纯本地向量化

优点：

- 无需新增外部 Embedding 配置
- 本地可直接运行
- 调试和测试成本低

缺点：

- 召回质量上限较低
- 后续升级时还需要补远端模式

### 方案二：纯远端 Embedding

优点：

- 检索效果通常更稳定
- 路径更接近正式线上方案

缺点：

- 配置复杂度和使用成本增加
- 依赖第三方接口，离线不可用
- 与当前 Demo 的“开箱即用”目标冲突

### 方案三：双模式

优点：

- 默认本地模式可直接工作
- 有条件时可平滑升级到远端 Embedding
- 更符合当前项目既要 Demo 可用、又要后续可扩展的目标

缺点：

- 需要额外维护两套向量提供器接口
- 测试矩阵略大

结论：采用方案三。

## 架构设计

RAG 功能新增 5 层职责：

1. 文件解析层
   负责读取上传文件内容，统一输出纯文本和元信息。

2. 文本切分层
   负责把长文本切成可检索片段，并保留片段顺序、字符区间和展示摘要。

3. 向量提供层
   统一定义 `embed_texts` 接口，底层可切换本地向量化或远端 Embedding。

4. 索引与检索层
   负责保存知识文件、片段和向量，执行 top-k 召回，并返回引用来源。

5. 聊天集成层
   负责把检索结果整合进现有会话消息构造逻辑，同时控制失败降级。

## 数据模型

### 本地文件存储

上传文件本体先落在工作目录下的本地数据目录，例如：

- `data/knowledge_files/<file_id>/<original_filename>`

原因：

- 当前项目仍是本地 Demo
- 避免把 PDF 二进制直接塞进 PostgreSQL
- 便于后续接入对象存储时做单点替换

### PostgreSQL 表

建议新增：

1. `knowledge_files`

- `id`
- `filename`
- `file_type`
- `storage_path`
- `content_hash`
- `file_size`
- `chunk_count`
- `embedding_mode`
- `created_at`
- `updated_at`

2. `knowledge_chunks`

- `id`
- `file_id`
- `chunk_index`
- `content`
- `preview`
- `embedding_vector`
- `char_start`
- `char_end`
- `created_at`

3. `session_knowledge_files`

- `id`
- `session_id`
- `file_id`
- `created_at`

设计说明：

- `embedding_vector` 初期以 JSON 文本存储，避免本阶段强依赖 `pgvector`。
- `content_hash` 用于避免重复上传完全相同的文件。
- 会话与文件是多对多关系，便于未来复用同一知识文件。

## 文档解析

### Markdown / TXT

- 直接按 UTF-8 读取
- 对空文件、不可解码文件给出稳定中文错误

### PDF

- 使用 Python PDF 文本抽取库提取纯文本
- 抽取失败时提示“PDF 暂无法解析文本”
- 不处理图片 OCR 和复杂版面恢复

选择标准：

- 优先使用依赖简单、稳定的纯 Python 方案
- 不依赖系统级 OCR 或桌面软件

## 文本切分

切分策略采用固定窗口加轻量重叠：

- 目标片段长度：约 `600` 到 `1000` 字符
- 重叠长度：约 `120` 到 `180` 字符
- 保证长文召回时上下文不断裂

切分输出字段：

- `chunk_index`
- `content`
- `preview`
- `char_start`
- `char_end`

规则：

- 去掉纯空白片段
- 预览内容只保留前若干字符，便于 UI 展示
- 单文件切分数量必须设上限，避免异常大文件拖垮页面

## 向量化策略

### 本地模式

本地模式采用轻量向量化：

- 用词项频次加归一化构造稀疏向量
- 以余弦相似度或点积完成检索

原因：

- 不新增大型机器学习依赖
- 方便本地运行和单元测试
- 对 Demo 级检索已足够

### 远端 Embedding 模式

如果用户在模型配置中额外提供 Embedding 相关配置，则优先调用远端 Embedding：

- Embedding API Key
- Embedding Base URL
- Embedding 模型名

聊天模型配置与 Embedding 配置分开，避免：

- 聊天模型不可用于 embedding
- 一个配置同时承担两种职责导致表单混乱

远端 Embedding 不可用时的回退策略：

- 索引构建失败时提示用户远端 Embedding 不可用
- 如果本地模式可用，则允许用户切回本地模式重新建索引
- 查询时不静默切换不同维度的旧索引，避免数据不一致

## 检索流程

用户提问时：

1. 判断当前会话是否启用知识库问答
2. 找到该会话绑定的知识文件
3. 生成查询向量
4. 召回 top-k 片段
5. 过滤低相似度结果
6. 拼接 RAG 上下文和引用来源
7. 调用现有聊天模型

拼接原则：

- 检索结果以单独的“参考资料”段落注入
- 不直接覆盖用户原问题
- 限制注入片段总长度，避免把现有上下文窗口挤爆

如果没有命中结果：

- 不报错
- 不强行插入空的知识库提示
- 直接走普通聊天链路

## 提示词集成

在现有系统提示词后追加一个固定结构的知识上下文块，例如：

- 告知模型“优先基于以下资料回答”
- 要求引用时尽量依据给定资料，不要编造文件内容
- 提示资料不足时明确说明

引用来源保存原则：

- 片段来源不写进用户消息
- 片段来源作为回答的附加元数据写入助手消息 `source` 或单独显示字段附近

如果发现现有 `ChatMessage` 结构不足以承载多条引用来源，则在本阶段扩展消息模型，但只扩到满足引用展示即可。

## UI 设计

知识库能力先放在设置弹窗的模型区域旁边或新增独立 tab，不额外开新页面。

本阶段 UI 需要支持：

- 上传一个或多个文件
- 查看当前会话已绑定文件
- 删除当前会话绑定的文件
- 开关“本会话启用知识库问答”
- 选择 Embedding 模式：本地 / 远端
- 展示最近一次回答的引用来源

展示策略：

- 文件列表展示文件名、类型、片段数
- 引用来源展示文件名 + 片段序号 + 片段摘要
- 不做复杂全文预览器

## 配置设计

新增环境变量建议：

- `RAG_MAX_FILE_SIZE_MB`
- `RAG_MAX_CHUNKS_PER_FILE`
- `RAG_TOP_K`

同时，模型配置页需要补充可选 Embedding 字段：

- `Embedding API Key`
- `Embedding Base URL`
- `Embedding 模型`

这些字段为空时，自动回退到本地向量模式。

## 错误处理

需要稳定处理并提示：

- 文件格式不支持
- 文件过大
- 文件解析失败
- 远端 Embedding 调用失败
- 索引构建失败
- 检索失败
- 本地索引与当前 Embedding 模式不兼容

要求：

- 用户看到的是稳定中文提示
- 日志记录更具体的错误类型
- 普通聊天能力不被 RAG 错误拖死

## 可观测性

在现有请求观测基础上，补充：

- 是否启用知识库问答

- 检索耗时
- Embedding 模式

这些信息先进入控制台日志，页面只展示轻量摘要，不做复杂分析面板。

## 风险与应对

### 风险一：PDF 文本质量不稳定

应对：

- 本阶段只承诺“抽取纯文本”
- 对失败文件给出清晰提示

### 风险二：本地向量效果一般

应对：

- 双模式保留远端 Embedding 升级路径
- 本地模式先满足开箱可用

### 风险三：页面逻辑继续膨胀

应对：

- 把知识库相关视图逻辑拆到单独模块
- 不在 [app.py](/Users/liuhuan/workspace/coding/backend/python/ai-chat-demo/app.py) 中直接堆积解析或检索逻辑

### 风险四：索引数据与配置不一致

应对：

- 在知识文件记录中保存 Embedding 模式
- 检索时校验模式兼容性
- 不兼容时给用户明确提示并要求重建索引

## 验收标准

满足以下条件即可认为本阶段完成：

- 用户可以上传 `Markdown / TXT / PDF`
- 当前会话可以绑定知识文件
- 提问时能够基于知识片段回答
- 页面可看到引用来源
- 远端 Embedding 未配置时，本地模式仍可工作
- 检索失败或无结果时，聊天功能仍可正常使用
- 有自动化测试覆盖主要链路
